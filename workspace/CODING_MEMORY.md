# Coding Memory Playbook

This is the practical layer that should make coding sessions feel continuous instead of amnesic.

## Goal

Before starting coding work, recover the most useful recent context cheaply:

- what project we were working on
- what files mattered recently
- what decisions were already made
- what blocker or next step was left unresolved

## Recall order

When a request looks like coding, debugging, implementation, refactor, repo work, PR work, CI investigation, or architecture follow-up, do this first:

1. Search durable memory and transcripts for the project/task.
2. Prefer the latest relevant daily memory notes and synthesized wiki/project notes.
3. Extract a short working set:
   - project
   - recent goal
   - files/components mentioned
   - blocker
   - next step
4. Only then start broad repo rediscovery.

## Working summary format

Keep coding recall summaries short and operational.

Example shape:

- Project: <name>
- Goal: <what we were trying to do>
- Recent files: <file list>
- Decisions: <important constraints or choices>
- Blocker: <if any>
- Next step: <best immediate continuation>

## Durable capture

When meaningful coding work finishes, or when stopping mid-stream, capture a compact note in the current daily memory file with:

- project/repo
- task
- important files touched
- decisions made
- blocker or risk
- next step

Do not dump long shell transcripts into durable memory. Capture only the reusable state.

## Wiki promotion rule

Promote coding notes into the wiki only when they become stable enough to matter beyond one session, for example:

- project overview
- recurring workflow
- architecture decision
- runbook
- durable environment fact

## What to avoid

- Do not use the wiki as a raw scratchpad for every coding turn.
- Do not force broad memory search when the task is obviously new and self-contained.
- Do not canonize tentative debugging guesses as durable facts.
